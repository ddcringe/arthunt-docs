# Обоснование выбора технологии асинхронного взаимодействия

## 1. Выбранное асинхронное взаимодействие

Отправка email-уведомлений о:

- новом **отклике специалиста на проект** заказчика (AH-MVP-007);
- новом **приглашении заказчика специалисту** (AH-MVP-006).

Взаимодействие является асинхронным по своей природе: в момент создания
отклика/приглашения пользователь уже получил HTTP-ответ `201 Created`, а
письмо должен получить адресат (заказчик или специалист) — в отдельном
процессе, без блокировки API и без потери при временной недоступности
SMTP-провайдера.

## 2. Сравнение рассмотренных технологий

| Технология | Подходит? | Комментарий |
|---|---|---|
| **RabbitMQ** | Да | Классический брокер сообщений для задач типа "fire-and-forget" с гарантией доставки. Нативные durable-очереди, DLX/DLQ, prefetch, manual ack. Умеет topic-роутинг для разных типов писем одним exchange. |
| Apache Kafka | Избыточно | Логи событий и большие потоки (millions/sec). Для ~300 уведомлений в месяц (см. оценку нагрузки в architecture-analysis.md) поднимать Kafka + Zookeeper/KRaft — неоправданная операционная сложность. Нет удобных встроенных повторов на конкретном сообщении. |
| NATS (JetStream) | Возможно, но | Лёгкий, но в выбранном стеке (Go + Gin, RU-облака) RabbitMQ — более распространённый выбор, с бОльшим числом готовых дашбордов, инструментов администрирования и опыта у команды. |
| WebSocket | Неприменимо | Решает задачу push-обновлений в открытом клиенте, а не доставки почты на SMTP-сервер. Не даёт надёжного хранения сообщений при офлайне получателя. |
| gRPC streams | Неприменимо | Подходит для синхронных двунаправленных потоков между сервисами, а не для буферизованной доставки писем через SMTP-провайдера. Не имеет персистентной очереди. |

## 3. Почему именно RabbitMQ

1. **Уже в стеке проекта.** В концепции (раздел 2.3 "Ограничения") и в
   существующей sequence-диаграмме AH-MVP-007 брокером сообщений
   явно выбран RabbitMQ. Решение не требует введения новой инфраструктуры.
2. **Характер взаимодействия — точечная доставка с гарантией.** Нужна
   именно очередь с персистентностью, подтверждениями и повторами, а не
   лог событий. RabbitMQ покрывает это из коробки (`durable=true`,
   `delivery_mode=2`, manual ack, DLX).
3. **Простая topic-маршрутизация.** Один exchange
   `arthunt.notifications` + routing-keys вида
   `notifications.email.<entity>.<event>` позволяет добавлять новые
   типы уведомлений (email/push/telegram) без изменения продьюсера —
   достаточно подписать новую очередь.
4. **Операционная совместимость с инфраструктурой.** RabbitMQ хорошо
   разворачивается в Yandex Cloud / VK Cloud (managed сервисы), имеет
   Prometheus-экспортёр и готовые Grafana-дашборды —
   соответствует требованию AH-NFR-006/007 (наблюдаемость и алерты).
5. **Устойчивость к пикам.** При временной недоступности SMTP письма
   остаются в очереди `email.notifications`, а после 3 неудачных попыток
   уходят в `email.notifications.dlq` — без потери данных и без
   блокировки основного API.

## 4. Топология (отражена в AsyncAPI)

```
 arthunt-api (producer)
       │  publish к exchange arthunt.notifications
       │      routing-key = notifications.email.response.created
       │      routing-key = notifications.email.invitation.created
       ▼
 [ exchange: arthunt.notifications, type=topic, durable ]
       │  bind: notifications.email.*
       ▼
 [ queue: email.notifications, durable, manual ack, x-dead-letter-exchange → arthunt.dlx ]
       │
       ▼
 notification-service (consumer) ── SMTP ──▶ email получателя
       │ (3 неудачных попытки → nack без requeue)
       ▼
 [ queue: email.notifications.dlq ]  ── Prometheus alert
```

## 5. Формат сообщения

JSON с общим конвертом (`envelope`) и бизнес-данными (`data`):

- `envelope.eventId` — UUID, используется потребителем как ключ
  идемпотентности (не отправлять дубль письма при повторной доставке).
- `envelope.schemaVersion` — версионирование контракта.
- `envelope.traceId` — сквозная трассировка (OpenTelemetry W3C).
- `data` — денормализованный минимум полей, достаточный для рендера
  письма без обязательного похода в PostgreSQL
  (recipient email/displayName, project title, actor displayName,
  deepLink). Это упрощает консьюмер и даёт устойчивость к изменениям
  данных между публикацией и доставкой.

Контракт описан в файле `asyncapi-notifications.yaml` (AsyncAPI 3.0).
